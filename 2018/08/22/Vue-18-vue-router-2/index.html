路由导航守卫

路由导航，即路由的前进后退跳转等，更明确的说，也就是路由改变时，进行的动作。
通过设置导航守卫，我们可以对路由导航进行编程式的控制，例如在离开页面时提示是否确定离开，在判断用户无token时跳转到登录页面等。

路由的导航守卫仅仅是路径参数或查询参数变化时，例如user/22导航到 user/33时是不会触发的。
对于这种变化，可以通过watch 组件$route页面路由对象或者通过组件的路由守卫beforeRouteUpdate来实现。

路由导航守卫分为全局的导航守卫，单个路由的导航守卫和组件内的守卫。

全局导航守卫

  • 全局导航守卫,注册在router实例上，有以下三个:
    • beforeEach 全局守卫
    • beforeResolve 全局解析守卫
    • afterEach 全局路由后置钩子

beforeEach((to, from, next) => {})
全局的路由导航前置守卫，在路由改变,导航触发时会进行异步解析(也就是通常所说的resolve过程)。

在导航守卫解析完毕之前，路由导航会一直处于等待之中。

当有多个路由守卫时，会按照创建顺序依次调用。

守卫方法会被传入如下三个参数:

  • to　 进入页面的页面路由对象route
  • from 离开页面的页面路由对象route
  • next　此参数的值为一个函数, 通过调用此函数来resolve守卫，使导航继续。根据next的参数不同，会有不同表现:
    • next() 无参数时，表示此守卫已解析完毕，进入下个导航钩子函数, 若无后续导航钩子，则导航完成。
    • next('/') 参数为一个路径或路径对象时，则跳转到对应路径。可接受与router.push同样类型的参数
    • next(false) 参数为false，终止导航，如果路径已经改变，则返回到from对应的页面。
    • next(error) 参数为Error类型，则导航终止同时触发router.onError

需要注意的是，在守卫的回调函数中必须确保调用next, 保证守卫被resolve。
尤其注意的是，在if逻辑判断的每个分支中都要调用。

router.onError 接受一个函数作为参数,当导航过程中出现错误时函数会被调用

beforeResolve((to, from , next) => {})
全局解析守卫，在所有组件内守卫和异步组件加载完毕之后,路由导航即将完成之前调用。
接收一个函数作为参数，传入函数的参数与beforeEach相同，为to,from，next。

afterEach((to, from) => {})
全局路由导航后置钩子，在路由导航完成之后会被调用。接收一个函数作为参数，传入函数的参数是没有next的，因为调用时导航已经是完成状态。

特定路由守卫

特定路由守卫只有一个:beforeEnter, 直接在路由配置上定义，类似于全局守卫beforeEach，但只作用于当前对应路由。

组件内守卫

组件内守卫有以下三个，用于守卫当前组件对用的路由:

  • beforeRouteEnter
  • beforeRouteUpdate

  • beforeRouteLeave
    使用方法类似于Vue组件的生命周期函数，均接受三个参数to,　from，next。
    可以在此守卫中获取页面的初始化数据，来代替一般使用的组件created声明周期函数和监听器获取数据的方式。详细可见我的另一篇博客中Vue页面初始数据获取的相关内容

    beforeRouteEnter(to, from, next)
    在进入所在组件的路由时调用。
    重要的一点是，此函数被调用时，路由还未导航完成，组件还处于未创建状态，是无法访问组件的this实例的。
    如果需要访问组件实例，可以传递一个参数为组件实例的回调函数到next中。

也只有此守卫的next方法可以接受回调函数作为参数。
对于其他两个导航守卫beforeRouteUpdate和beforeRouteLeave来说，因为可以访问this，所以不能也不需要接受一个参数为组件实例的回调函数来访问组件。

beforeRouteUpdate(to, from, next)
在当前路由发生了变化，但并未重新渲染组件，而是复用当前组件时被调用。
在开篇我们提到，其它的路由导航守卫是不响应路由路径参数和查询参数的变化的。
类似于user/22到user/33这种路由变化，组件会被复用,　进而导致我们修改路由后组件的数据并不会重新响应和获取。
如果我们想监测路由路径参数或查询参数变化，进而重新获取数据或者进行一些操作，可以通过此守卫来实现。
当然，通过watch页面的路由对象$route，也可以达到相同的效果。

beforeRouteLeave(to, from, next)
当离开此组件所对应路由，导航向其他路由时触发。
可以在此函数中进行未保存提示等操作。
可以通过在守卫中调用next(false)来取消导航，停留在当前组件。

路由守卫的调用顺序

在一次完整的导航中，各路由守卫的触发和调用顺序如下:

  1. 导航触发
  2. 在离开的组件中调用beforeRouteLeave
  3. 调用全局路由守卫beforeEach
  4. 调用特定路由配置守卫beforeEnter
  5. 导入异步组件
  6. 在激活的组件中调用beforeRouterEnter
  7. 调用全局解析守卫beforeResolve
  8. 导航完成
  9. 调用全局导航后置钩子afterEach123232
  10. 调用传递给beforeRouteEnter的next函数的回调函数

路由模式mode

Vue Router支持三种模式的URL, 分别为:

  • hash模式(浏览器环境下默认模式
  • history模式
  • abstract模式（node.js环境下默认模式
  • 可以在vue-router的配置中定义使用何种模式:

hash模式

即通过URL的hash来在单页面应用中模拟 URL的模式。类似URL如下:

hash模式背后的原理是浏览器的onhashchange事件，通过检测URL的哈希变化，来实现前端路由。
对于哈希模式来说，发送到后端的请求是始终不会包含哈希之后的前端路径内容的。项目的路由路径匹配，是始终在前端进行的。

histroy模式

通过HTML5中提供的history api，前端路由也可以拥有完全的路径自由，不必再在路径中添加丑陋的哈希了。

HTML History API 为开发人员提供了在不刷新整个页面下修改当前URL的功能。
有以下几个API:

  • froward 前进
  • back　后退
  • go 跳转
  • pushState 保存历史状态, 接收三个参数，URL对应的状态数据state，页面title，和 URL
  • replaceState 替换历史状态

在histroy模式中，URL表现与正常的非SPA项目基本完全一致。
但与hash模式相比，存在一个问题，在此模式下，页面刷新时，会向服务器发起请求，但服务器并没有对应的路由，就会出现404错误,因此需要在后端进行相应的配置：对于任何的与前端项目有关的路径，都应该返回前端项目的根目录路径，从而让前端能够接管到路由并进行匹配。

滚动

在导航到某个组件时，可以通过Vue Router的滚动行为来控制页面滚动到某个位置。
通过在路由配置项中定义scrollBehavior方法，来在路由切换时自定义控制页面滚动。
scrollBehavior接受三个参数，to和from为进入和离开页面的页面路由对象。
savedPosition应用于页面通过记录原生的浏览器前进后退按钮导航时的滚动位置，通过返回此值，可以模拟浏览器原生的滚动行为。

懒加载

懒加载，也就是通常所说的路由异步加载，对于某些非核心组件，只在需要组件时进行异步加载，从而提高项目的首屏加载速度。
在Vue　Router中，使用异步加载是非常简单的，我们只需要将组件设置为异步组件，当路由对应到此组件时，就会异步加载。

同时我们还可以通过WebPack的命名chunk, 来将一些异步组件打包到同一个块中，在路由加载其中一个时，就会获取其他所有的异步组件，从而更好的组织我们的路由组件的加载。

router-link标签

在上一篇博客中我们提到在HTML中，我们可以通过<router-link></router-link>标签来进行路由导航。
相比于a标签，router-link标签对Vue Router的路由跳转兼容性和配合性更好，同时它还可以接受一些属性来达到功能上的增强:

  • to
    在上篇博客的路由导航中介绍过，用于设置跳转对象，相当于a标签的href属性，不同之处在于除了可以接受路径字符串，还可以接受vue-router路径对象。
  • replace
    布尔值属性，当添加此属性时，在点击router-link标签时，会调用路由器router的replace方法进行导航，而不是push
  • append
    布尔值属性，当添加此属性时，则将当前页面路径作为根路径进行跳转。
  • tag
    指定router-link标签在真实HTML中渲染出的标签。可以为li标签，span标签等。在router-link中有slot内容时，非常实用。
  • active-class
    当前标签对应路径匹配路由跳转路径时自动添加的CSS类名。
  • exact
    布尔值属性，是否关闭路径的全包含匹配,默认是开启的，也就是说，只要路由跳转路径中包含当前标签的对应跳转路径，无论是否完全对应，都会为标签添加active-class指定的类名。
    使用exact属性，则只有路由导航跳转路径完全匹配当前标签对应跳转路径时，才会添加类名。
  • exact-active-class
    当前标签对应路径完全匹配路由跳转路径时自动添加的CSS类名。
  • event
    用来触发导航的事件名。通常标签都是通过click来触发路由导航，当然也可以通过其他事件来触发，不过不常见，用户体验较差。

可以在Router的构建选项中设置router-link标签的的全局默认active-class和exact-active-class。

如果需要单个路由的过渡效果，直接在路由对应的组件内使用transtion标签即可。

关于Vue Router的介绍就到这里，Vue 官方的这个路由库设计的还是非常优雅和好用的。如果有想深入研究一下SPA应用路由处理的同学，从这个库入手，是非常不错的选择。
